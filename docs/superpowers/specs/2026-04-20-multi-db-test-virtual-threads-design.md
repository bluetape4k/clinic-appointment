# Multi-DB Test Support + Virtual Threads API Design Spec

**Date**: 2026-04-20
**Status**: Draft - 승인 대기

---

## 1. 개요 및 목표

### 배경
현재 `appointment-core` 모듈의 도메인 테스트는 H2 인메모리 DB에서만 실행된다.
`appointment-api`는 Flyway migration만 H2/PostgreSQL/MySQL matrix로 검증하고, 통합 테스트는 단일 DB로 실행된다.
프로덕션에서 PostgreSQL/MySQL을 사용하므로, DDL 호환성 및 쿼리 동작 차이를 CI 레벨에서 조기 발견해야 한다.

### 목표
1. **appointment-core**: H2, PostgreSQL, MySQL8 3개 DB에서 도메인 테스트 실행
2. **appointment-api**: Spring Profile 기반 DB 선택 (`test`, `test-postgresql`, `test-mysql`)
3. **GitHub CI**: 3개 DB 테스트를 matrix strategy로 병렬 실행
4. **Virtual Threads**: Spring Boot 표준 설정으로 Virtual Threads 활성화

### 현재 코드베이스와의 정렬
- `appointment-api`에는 이미 `application-postgresql.yml`과 `application-flyway.yml`이 존재한다. 따라서 PostgreSQL/Flyway는 신규 도입이 아니라 기존 구성 확장으로 다룬다.
- `appointment-api`의 Flyway 3종 검증(H2/PostgreSQL/MySQL)은 이미 CI에 존재한다. 본 스펙의 CI 변경은 기존 Flyway matrix를 유지하면서 API 통합 테스트 matrix를 추가하는 방향이어야 한다.
- `appointment-api`는 `exposed_spring_boot4_starter`를 이미 사용 중이므로, 별도 `Database` 빈을 추가하는 설계는 기본안으로 채택하지 않는다.
- `appointment-api`의 `SchemaInitConfig`는 `dev,test` 프로파일에서 `SchemaUtils.create(...)`를 수행하므로, PostgreSQL/MySQL 기반 테스트 프로파일과 함께 사용할 때 Flyway와 충돌하지 않도록 `@ConditionalOnProperty(spring.flyway.enabled=false)`를 추가하여 Flyway 비활성 시에만 동작하도록 제한한다.

## 2. 문제 재정의 및 제약 확인

### 2.1 R2DBC 제거 — 순수 JDBC로 통일

**현재 상태**: `appointment-core`의 모든 테스트 코드는 JDBC `transaction {}` 블록 기반이다.
`exposed_r2dbc`, `bluetape4k_exposed_r2dbc`, `bluetape4k_exposed_r2dbc_tests`, `r2dbc_h2` 의존성이 있으나
실제 사용되는 코드가 없다.

**결론**: R2DBC 관련 의존성을 전부 제거하고 순수 JDBC로 통일한다.
- **제거**: `exposed_r2dbc`, `bluetape4k_exposed_r2dbc`, `kotlinx_coroutines_reactor` (main)
- **제거**: `bluetape4k_exposed_r2dbc_tests`, `r2dbc_h2` (test)
- **추가**: `bluetape4k_exposed_jdbc_tests` + Testcontainers JDBC 드라이버 (test)

### 2.2 Virtual Threads + Exposed JDBC 호환성

**분석**: Java 25의 Virtual Threads는 blocking I/O를 자동으로 unmount하여 carrier thread를 반환한다.
JDBC는 blocking I/O이므로 Virtual Threads와 매우 궁합이 좋다 -- JDBC 호출 시 virtual thread가 park되고,
carrier thread는 다른 virtual thread를 실행한다.

**주의점**: `synchronized` 블록 내에서의 blocking은 carrier thread를 pin할 수 있다.
HikariCP 4.x는 이미 Virtual Threads를 고려하여 `synchronized` 대신 `ReentrantLock`을 사용한다.
Exposed JDBC의 transaction manager도 `ThreadLocal` 기반이라 virtual thread별로 격리된다.

**결론**: Exposed JDBC + Virtual Threads는 안전하다. 추가 설정 불필요.

### 2.3 H2 MySQL Mode vs 실제 MySQL8

| 영역 | H2 `MODE=MySQL` | MySQL 8.4 |
|------|-----------------|-----------|
| AUTO_INCREMENT | `BIGINT AUTO_INCREMENT` | `BIGINT AUTO_INCREMENT` |
| IDENTITY | 미지원 | 미지원 (PostgreSQL 전용) |
| JSON 함수 | 제한적 | 완전 지원 |
| Window 함수 | 기본 지원 | 완전 지원 |
| Charset/Collation | 무시됨 | utf8mb4/utf8mb4_bin 필요 |
| Flyway vendor | `h2` 디렉토리 | `mysql` 디렉토리 |

**결론**: H2 MySQL Mode는 빠른 스모크 테스트용이고, DDL/쿼리 정합성은 실제 MySQL8 Testcontainer에서 검증해야 한다.
이미 Flyway migration은 별도 SQL 파일로 분리되어 있어 (`db/migration/h2/`, `db/migration/postgresql/`, `db/migration/mysql/`) 호환성 문제 없다.

## 3. 리스크 및 실패 모드

### Risk 1: Testcontainers Docker 의존성으로 인한 CI 불안정
- **영향**: Docker daemon 미응답 시 PostgreSQL/MySQL 테스트 전체 실패
- **확률**: 중 (GitHub Actions Ubuntu runner에서 간헐적 발생)
- **완화**: `fail-fast: false` + H2 테스트는 항상 Testcontainers 없이 실행
- **완화**: `TESTCONTAINERS_RYUK_DISABLED=true` 설정 (이미 CI에서 사용 중)

### Risk 2: appointment-core 테스트의 DDL 호환성 깨짐
- **영향**: `SchemaUtils.create()` + `MigrationUtils.statementsRequiredForDatabaseMigration()`이 DB별로 다른 DDL 생성
- **확률**: 높음 (H2 `AUTO_INCREMENT` vs PostgreSQL `GENERATED BY DEFAULT AS IDENTITY`)
- **완화**: `SchemaUtils.createMissingTablesAndColumns()` 사용 (DB dialect에 맞게 자동 변환)
- **완화**: Exposed의 Table DSL은 dialect별 DDL을 자동 생성하므로, core 테스트에서는 SchemaUtils에 위임

### Risk 3: MySQL8의 대소문자 민감도 차이
- **영향**: MySQL `lower_case_table_names` 설정에 따라 테이블명 해석이 달라짐
- **확률**: 낮음 (Exposed가 테이블명을 자동 처리)
- **완화**: Testcontainers MySQL8 기본 설정 사용 (Linux = `lower_case_table_names=0`)

### Risk 4: CI 실행 시간 증가
- **영향**: 3x DB matrix로 총 CI 시간 30-50% 증가
- **확률**: 확실
- **완화**: matrix 병렬 실행 + `fail-fast: false`로 독립 실행
- **완화**: H2 테스트는 1-2분 이내, Testcontainers 테스트는 컨테이너 기동 포함 3-5분

### Risk 5: Virtual Threads + Testcontainers 상호작용
- **영향**: Testcontainers 내부에서 `synchronized`를 사용할 경우 virtual thread pinning 발생 가능
- **확률**: 매우 낮음 (Testcontainers는 테스트 셋업 시에만 사용, 런타임 API 실행 경로가 아님)
- **완화**: Virtual Threads는 API 런타임에만 적용, 테스트 인프라는 영향 없음

## 4. 접근 방식 비교

### 접근법 A: bluetape4k `exposed-jdbc-tests` TestDB 방식 (ParameterizedTest)  **[채택]**

**개요**: `AbstractExposedTest` + `TestDB` enum + `withTables()` 헬퍼를 그대로 활용.
각 테스트 메서드에 `@ParameterizedTest @MethodSource(ENABLE_DIALECTS_METHOD)`를 붙여 H2/PG/MySQL을 순회.

`enabledDialects()`는 `TestDBConfig.useFastDB`(기본값 `false`)에 따라 `{H2, POSTGRESQL, MYSQL_V8}`을 반환하며,
`withDb()`는 `TestDB.db`를 캐시하고 세마포어로 직렬화하여 같은 Testcontainer를 테스트 간 재사용한다.

**장점**:
- bluetape4k 표준 패턴 그대로 활용 — 팀 내 일관성
- `withTables(testDB, *tables) { ... }` 한 줄로 DB 전환 + 테이블 생성/정리
- 로컬 개발 시 `TestDBConfig.useFastDB = true`로 H2만 빠르게 실행 가능
- 커스텀 `TestDatabaseFactory` 클래스 불필요
- CI matrix 없이 단일 Job에서 3개 DB 모두 검증

**단점**:
- 기존 11개 테스트 파일 전체 리팩토링 필요 (`Database.connect` 직접 호출 → `withTables` 전환)
- 각 테스트 메서드에 `testDB: TestDB` 파라미터가 추가되어 시그니처 변경
- 단일 JVM에서 3개 DB를 순차 실행 (접근법 B의 CI 완전 병렬보다 느림)

**채택 이유**: bluetape4k 표준 패턴 준수, `withTables()` 재사용, Testcontainer 캐시로 효율적.

### 접근법 B: Gradle System Property + CI matrix (DB 선택을 외부에서 주입)

**개요**: Gradle `test` 태스크에 `-Dtest.db=h2|postgresql|mysql` 시스템 프로퍼티를 주입.
테스트 베이스 클래스가 이 프로퍼티를 읽어 `Database.connect()`에 전달할 JDBC URL/driver를 결정.
CI에서는 matrix strategy로 3개 DB를 별도 Job으로 병렬 실행.

**장점**:
- 기존 테스트 코드 최소 변경 -- `Database.connect(...)` 한 줄만 팩토리 메서드로 교체
- CI matrix로 3개 DB가 완전 병렬 실행 (총 시간 = 가장 느린 DB 시간)

**단점**:
- 커스텀 `TestDatabaseFactory` 클래스 작성 필요 (약 60줄)
- `bluetape4k_exposed_jdbc_tests`의 `withTables` 헬퍼를 활용하지 않아 표준 패턴 이탈

**기각 이유**: bluetape4k 표준 패턴(`AbstractExposedTest` + `withTables`)을 활용할 수 있는 상황에서 커스텀 팩토리를 별도 작성하는 것은 불필요한 중복이다.

### 접근법 C: 하이브리드 (Core는 TestDB enum, API는 Profile)

**개요**: `appointment-core`는 접근법 A, `appointment-api`는 접근법 B.

**장점**: 각 모듈에 최적화된 방식 사용

**단점**: 모듈 간 테스트 패턴 불일치, 유지보수 부담 증가

**기각 이유**: 패턴 불일치는 장기적 기술 부채.

## 5. 아키텍처 결정

| 결정 | 선택 | 근거 |
|------|------|------|
| Core DB 전환 방식 | `AbstractExposedTest` + `@ParameterizedTest` + `withTables()` | bluetape4k 표준 패턴, `withTables()` 재사용, Testcontainer 캐시 |
| API DB 전환 방식 | Spring Profile (`test`, `test-postgresql`, `test-mysql`) + 기존 `application-postgresql.yml`/`application-flyway.yml` 재사용 | 현재 코드와 정렬되는 Spring Boot 표준 방식 |
| Testcontainers 관리 | bluetape4k singleton 패턴 (`MySQL8Server.Launcher.mysql`, `PostgreSQLServer.Launcher.postgres`) | JVM 내 1회 기동, 테스트 간 재사용 |
| CI 전략 | Matrix strategy (`db: [h2, postgresql, mysql]`) | 3개 DB 완전 병렬 |
| Virtual Threads | `spring.threads.virtual.enabled=true` | Spring Boot 표준 설정, 커스텀 Tomcat customizer 불필요 |
| Exposed Database 설정 | 기본은 starter auto-config 유지, 필요 시 `DatabaseConfig`만 최소 커스터마이징 | 중복 `Database` 빈 생성으로 인한 충돌 방지 |
| HikariCP 사용 | 프로덕션에서는 Spring Boot auto-config, 테스트에서는 Testcontainers DataSource 사용 | 현재 구성과 자연스럽게 통합 |

### 6.2 로컬 테스트 지원 모듈 — exposed-workshop 패턴 복사

`bluetape4k-exposed-jdbc-tests`의 `TestDBConfig`는 외부 라이브러리이고 시스템 프로퍼티 기반 초기화를 지원하지 않는다 (`var useFastDB = false` 하드코딩).
대신 `exposed-workshop/00-shared/exposed-shared-tests`에서 이미 검증된 패턴을 `appointment-core` 테스트 소스로 복사한다.

**복사 대상 파일 (5개):**

| 원본 (`exposed.shared.tests`) | 복사 위치 (`io.bluetape4k.clinic.appointment.test`) | 역할 |
|-------------------------------|------------------------------|------|
| `TestDB.kt` | `appointment-core/src/test/.../test/TestDB.kt` | DB enum + `useFastDB`/`useDB` 시스템 프로퍼티 읽기 |
| `AbstractExposedTest.kt` | `.../test/AbstractExposedTest.kt` | `enableDialects()`, `ENABLE_DIALECTS_METHOD` |
| `WithTables.kt` | `.../test/WithTables.kt` | `withTables(testDB, *tables) { ... }` |
| `WithDb.kt` | `.../test/WithDb.kt` | Semaphore 기반 직렬화, DB 캐시 |
| `Containers.kt` | `.../test/Containers.kt` | PostgreSQL/MySQL8 Testcontainers 싱글톤 |

**핵심 — `TestDB.kt` line 22 (확인된 코드):**
```kotlin
// JVM 시작 시 시스템 프로퍼티를 읽어 초기화 (가정 아님)
val useFastDB: Boolean = System.getProperty("exposed.test.useFastDB", "false").toBoolean()

fun enabledDialects(): Set<TestDB> {
    val useDB = System.getProperty("exposed.test.useDB")
    if (!useDB.isNullOrBlank()) {
        return useDB.split(",").map { it.trim() }
            .mapNotNull { name -> entries.find { it.name.equals(name, true) } }
            .toSet().ifEmpty { setOf(H2) }
    }
    return if (useFastDB) setOf(H2) else setOf(H2, POSTGRESQL, MYSQL_V8)
}
```

패키지만 `exposed.shared.tests` → `io.bluetape4k.clinic.appointment.test`로 변경 후 그대로 사용한다.

**테스트 클래스 적용:**

### 6.3 기존 테스트 변경 패턴

**Before**:
```kotlin

@BeforeAll

@Test

    transaction(db) {

    }
}
```

**After**:
```kotlin
class SlotCalculationServiceTest: AbstractExposedTest() {

    @ParameterizedTest
    @MethodSource(ENABLE_DIALECTS_METHOD)
    fun `가용 슬롯 계산`(testDB: TestDB) = withTables(
        testDB, Clinics, Doctors, DoctorSchedules, Appointments
    ) {
        // 테스트 코드 — transaction 내부에서 실행됨
```

변경 포인트:
1. `AbstractExposedTest` 상속 추가
2. `@Test` → `@ParameterizedTest @MethodSource(ENABLE_DIALECTS_METHOD)` + `testDB: TestDB` 파라미터
3. `@BeforeAll` / `Database.connect(...)` 제거 → `withTables()` 내부에서 처리
4. `SchemaUtils.create()` / `deleteAll()` 제거 → `withTables()` 가 drop → create → 블록 실행 → drop 수행

### 6.4 테스트 DB 제어 — Gradle 프로젝트 프로퍼티 방식

`TestDBConfig`를 코드에서 직접 수정하지 않고, Gradle 프로젝트 프로퍼티로 제어한다.
`build.gradle.kts`에서 프로퍼티를 읽어 시스템 프로퍼티로 테스트에 전달한다.

**appointment-core/build.gradle.kts** 추가:
```kotlin
tasks.withType<Test>().configureEach {
    val useFastDB = (project.findProperty("useFastDB") as? String)?.toBoolean() ?: false
    systemProperty("exposed.test.useFastDB", useFastDB)

    val useDB = (project.findProperty("useDB") as? String) ?: ""
    if (useDB.isNotBlank()) {
        systemProperty("exposed.test.useDB", useDB)
    }
}
```

**TestDBConfig** 시스템 프로퍼티 읽도록 초기화:
```kotlin
object TestDBConfig {
    val useFastDB: Boolean = System.getProperty("exposed.test.useFastDB", "false").toBoolean()
    val useTestcontainers: Boolean = System.getProperty("exposed.test.useTestcontainers", "true").toBoolean()

    // useDB 프로퍼티: "H2,PostgreSQL,MySQLV8" 형태로 DB 목록 지정 (비어있으면 enabledDialects() 기본값 사용)
    val specifiedDBs: Set<String> = System.getProperty("exposed.test.useDB", "")
        .split(",")
        .map { it.trim() }
        .filter { it.isNotBlank() }
        .toSet()
}
```

**사용 예시:**
```bash

./gradlew :appointment-core:test -PuseFastDB=true

# PostgreSQL + MySQL8만 실행 (Testcontainers 필요)
./gradlew :appointment-core:test -PuseDB=PostgreSQL,MySQLV8

# H2 + PostgreSQL + MySQL8 모두 실행 (기본값, CI 환경)
./gradlew :appointment-core:test
```

**CI 실행 (기본값 그대로):**

```yaml
    TESTCONTAINERS_RYUK_DISABLED: "true"
```

> **주의**: 6.2에 기술된 로컬 `TestDB`는 JVM 시작 시 시스템 프로퍼티를 읽으므로,

## 7. appointment-api 변경 사항

### 7.1 Exposed 설정 전략

`appointment-api`는 이미 `exposed_spring_boot4_starter`를 사용 중이므로, 기본안은 starter auto-config를 유지한다.
즉, **새로운 `Database` 빈을 직접 등록하지 않는다.**

필요한 경우에만 다음처럼 `DatabaseConfig` 성격의 최소 커스터마이징만 추가한다.

```kotlin
@Configuration(proxyBeanMethods = false)
class ExposedDatabaseConfig {
    @Bean
    fun databaseConfig(): DatabaseConfig = DatabaseConfig {
        useNestedTransactions = true
    }
}
```

> [!warning]
> `Database.connect(dataSource, ...)`를 직접 빈으로 등록하면 `exposed_spring_boot4_starter`의 자동 설정과 역할이 겹칠 수 있다. 기본 설계에서는 이를 피한다.

### 7.2 Virtual Threads 설정

Spring Boot 표준 설정을 우선 사용한다. 별도 `TomcatVirtualThreadConfig`는 기본안에서 제외한다.

**application.yml** (기본 - H2):
```yaml
spring:
    application:
        name: appointment-api
    datasource:
        url: jdbc:h2:mem:appointment;DB_CLOSE_DELAY=-1
        driver-class-name: org.h2.Driver
    flyway:
        enabled: false
        locations: classpath:db/migration/{vendor}
    threads:
        virtual:
            enabled: true

server:
    port: 8080
```

> [!note]
> 별도 Tomcat customizer는 Tomcat 전용 세밀한 제어 또는 명시적 로깅이 꼭 필요할 때만 검토한다. 기본 설정만으로 Virtual Threads 활성화가 가능하다.

### 7.3 프로파일별 설정 정리

**기존 파일 재사용:**
- `application.yml`: 기본 H2
- `application-postgresql.yml`: 이미 존재, 재사용
- `application-flyway.yml`: 이미 존재, 재사용

**신규 파일:**
- `application-mysql.yml`
- `application-test-postgresql.yml`
- `application-test-mysql.yml`

**application-postgresql.yml**은 기존 값을 최대한 유지하고, 필요 시 계정 정보만 테스트 환경과 일치시킨다.

**application-mysql.yml** (신규):
```yaml
spring:
    datasource:
        url: jdbc:mysql://localhost:3306/appointment?useSSL=false&characterEncoding=UTF-8&serverTimezone=UTC
        driver-class-name: com.mysql.cj.jdbc.Driver
        username: test
        password: test
    flyway:
        enabled: true
        locations: classpath:db/migration/mysql
        baseline-on-migrate: true
```

### 7.4 테스트 프로파일 설정

**application-test.yml**은 H2 기본 테스트용으로 유지한다.

**application-test-postgresql.yml** (신규):

### 7.5 SchemaInitConfig 프로파일 분리

현재 `SchemaInitConfig`는 `dev,test` 프로파일에서 `SchemaUtils.create(...)`를 수행한다.
이 상태에서 `test-postgresql` 또는 `test-mysql`과 함께 실행하면 Flyway와 스키마 생성 책임이 충돌할 수 있다.

**채택 방식: `@ConditionalOnProperty(flyway.enabled=false)` 추가**

프로파일명 변경 없이, Flyway가 비활성화된 경우에만 SchemaInitConfig가 동작하도록 조건을 추가한다.

**변경 전** (`appointment-api/src/main/kotlin/.../config/DatabaseConfig.kt:38`):
```kotlin
@Configuration(proxyBeanMethods = false)
@Profile("dev", "test")
class SchemaInitConfig {
```

**변경 후**:
```kotlin
@Configuration(proxyBeanMethods = false)
@Profile("dev", "test")
@ConditionalOnProperty(name = ["spring.flyway.enabled"], havingValue = "false", matchIfMissing = true)
class SchemaInitConfig {
```

**동작 결과**:
| 환경 | `flyway.enabled` | SchemaInitConfig 활성화 |
|------|-----------------|------------------------|
| `dev` (H2) | `false` (기본값) | ✅ 활성화 |
| `test` (H2, `application-test.yml`) | `false` | ✅ 활성화 |
| `test-postgresql` + `application-flyway.yml` | `true` | ❌ 비활성화 (Flyway가 담당) |
| `test-mysql` + `application-flyway.yml` | `true` | ❌ 비활성화 (Flyway가 담당) |

기존 테스트의 `@ActiveProfiles("test")` 어노테이션은 변경할 필요가 없다.

### 7.6 TestcontainersDatabaseInitializer (신규)

Spring Boot 테스트에서 Testcontainers를 DynamicPropertySource로 연결:

```kotlin
abstract class AbstractApiIntegrationTest {
    companion object {
        @JvmStatic
        @DynamicPropertySource
        fun configureProperties(registry: DynamicPropertyRegistry) {
            val profile = System.getProperty("spring.profiles.active", "test")
            when {
                profile.contains("postgres") -> {
                    val pg = PostgreSQLServer.Launcher.postgres
                    registry.add("spring.datasource.url") { pg.jdbcUrl }
                    registry.add("spring.datasource.username") { pg.username }
                    registry.add("spring.datasource.password") { pg.password }
                }
                profile.contains("mysql") -> {
                    val mysql = MySQL8Server.Launcher.mysql
                    registry.add("spring.datasource.url") { mysql.jdbcUrl }
                    registry.add("spring.datasource.username") { mysql.username }
                    registry.add("spring.datasource.password") { mysql.password }
                }
            }
        }
    }
}
```

## 8. GitHub CI 변경 사항

### 8.1 test-core Job — 변경 없음 (단일 Job 유지)

`@ParameterizedTest`로 단일 Job 내에서 H2 → POSTGRESQL → MYSQL_V8을 순차 실행하므로 matrix 불필요.

```yaml
test-core:
    name: Test / Core & Event
    runs-on: ubuntu-latest
    needs: build
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with:
          java-version: ${{ env.JAVA_VERSION }}
          distribution: ${{ env.JAVA_DISTRIBUTION }}
      - uses: gradle/actions/setup-gradle@v4
        with:
          gradle-version: wrapper
          cache-read-only: true
      - name: Test core & event (H2 + PostgreSQL + MySQL8)
        run: |
          ./gradlew \
            :appointment-core:test \
            :appointment-event:test \
            --parallel
        env:
          GRADLE_OPTS: "-Dorg.gradle.daemon=false"
          TESTCONTAINERS_RYUK_DISABLED: "true"
```

### 8.2 test-api Job 변경

**Before**: 단일 실행
**After**: Matrix strategy
```yaml
test-api:
    name: Test / API (${{ matrix.db }})
    runs-on: ubuntu-latest
    needs: build
    strategy:
      fail-fast: false
      matrix:
        include:
          - db: h2
            spring-profile: test
          - db: postgresql
            spring-profile: test,test-postgresql
          - db: mysql
            spring-profile: test,test-mysql
    steps:
      - ...
      - name: Test API (${{ matrix.db }})
        run: |
          ./gradlew :appointment-api:test \
            -Dspring.profiles.active=${{ matrix.spring-profile }} \
            --no-daemon
        env:
          GRADLE_OPTS: "-Dorg.gradle.daemon=false"
          TESTCONTAINERS_RYUK_DISABLED: "true"
```

### 8.3 Artifact 이름

**`test-core` Job** (matrix 없음 — 단일 실행):
```yaml
- name: Upload test results
  uses: actions/upload-artifact@v4
  with:
    name: test-results-core          # ← matrix 없으므로 suffix 불필요
    path: appointment-core/build/reports/tests
- name: Upload coverage
  uses: actions/upload-artifact@v4
  with:
    name: coverage-core              # ← 고정 이름
    path: appointment-core/build/reports/jacoco
```

**`test-api` Job** (matrix job — `${{ matrix.db }}` 접미사 필수):
> ⚠️ matrix job에서 동일한 artifact 이름을 사용하면 CI 실패. 반드시 `${{ matrix.db }}` 접미사 추가.

```yaml
- name: Upload test results
  uses: actions/upload-artifact@v4
  with:
    name: test-results-api-${{ matrix.db }}   # ← matrix.db 접미사 필수
    path: appointment-api/build/reports/tests
- name: Upload coverage
  uses: actions/upload-artifact@v4
  with:
    name: coverage-api-${{ matrix.db }}        # ← matrix.db 접미사 필수
    path: appointment-api/build/reports/jacoco
```

### 8.4 coverage-report Job 업데이트

현재 `coverage-report` Job은 모듈별 단일 실행 결과를 전제로 집계한다. `test-core`/`test-api`를 matrix로 전환하면 다음을 함께 검토해야 한다.

- matrix별 coverage artifact 이름 분리
- 후속 집계 Job에서 `${{ matrix.db }}` suffix를 가진 artifact를 모두 다운로드하도록 변경
- DB별 개별 리포트만 보존할지, 병합 리포트를 만들지 결정

즉, matrix 전환의 영향 범위는 `ci-status`뿐 아니라 `coverage-report`까지 포함된다.

### 8.5 ci-status Job 업데이트

`needs` 목록에서 `test-core`와 `test-api`가 matrix로 변경되어도 GitHub Actions는 자동으로 모든 matrix 조합 완료를 기다린다. 기본적으로 추가 변경은 불필요하다.

## 9. Virtual Threads 상세

### 9.1 활성화 조건

- `spring.threads.virtual.enabled=true`
- Java 21+ 런타임 필수 (현재 Java 25 사용)
- Spring Boot가 Virtual Threads를 지원하는 환경이어야 함

### 9.2 영향 범위

| 컴포넌트 | Virtual Threads 영향 |
|----------|---------------------|
| Tomcat 요청 처리 | Virtual thread per request (기존: platform thread pool) |
| Exposed JDBC transaction | Virtual thread에서 blocking I/O → 자동 unmount |
| HikariCP connection pool | Virtual thread-safe (ReentrantLock 기반) |
| Spring Security filter chain | Virtual thread에서 실행 |
| Timefold Solver | 별도 thread pool 사용, 영향 없음 |

### 9.3 비활성화 방법

```yaml
spring:
    threads:
        virtual:
            enabled: false
```
